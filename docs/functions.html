This page

This page details the functionality and methods used in the program. It is intended to provide enough detail to understand the thinking and progression of the program, or to enable a user to extract or re-purpose individual methods or classes.

Inputs

Parameter input

The program commences with a call to test_initialiser(.) which takes the following arguments:

start_nodes an integer or set of integers (space separated) specifying the origin or start nodes for which to compute from
start_index the starting index of the adjacency list, either 0 or 1
input_path the file path to the adjacency list (a text file)
user_specified_write_results Boolean to write the resulting matrix/ matrices to a JSON and subnetwork adjacency list to a text file
output_dir the output directory for the results to be written to (if user_specified_write_results is TRUE, else output_dir can be set NULL)

Note that this program assesses single origin paths only. While many origin nodes can be inputted, each one is assessed individually and independently of the other specified origin nodes.

Command Line Interface

Note

The current implementation does not take an input for the output directory (output_dir). Instead the directory of the input file is used by default. Users wishing to specify a different output directory should use a direct function call.

The function test_initialiser(.) can be called directly, but it is expected most users will initialise the program using the Command Line Interface (CLI) which collects the user’s responses and passes them to test_initialiser(.). Under normal execution, the CLI is called from the main() function which will collect the user’s inputs using helper functions which enforce input class requirements and excepts any invalid inputs.

Input adjacency list

The input path for the program should point to a text file containing the adjacency list for the network of interest. The adjacency list should contain the network links, one per line, in the format [origin destination cost]. The nodes and costs must be space seperated. An example from Test Graph A is presented below:

0 1 1
1 2 1
1 2 2
1 2 3
2 3 1

Note

Only static costs can be considered by this program. It is not currently possible to consider dynamic costs such as those used for route assignment (e.g. The Bureau of Public Roads (BPR) function). It is recommended that the user use the free flow travel time or segment length if a free flow time is not available for a network.

Data types

Once the input files are read they are stored in a the custom data structure network_map which contain the data class Edge. See the Data Structures page for details on the implementation and structure of these data structures.

RoadNetwork Class

Once initialised, the program executes its core functionality using the RoadNetwork Class which contains the network information and functions to complete the program computations. The following sections contain an overview of the RoadNetwork Class functions.

sssp_dijkstra_solver(.)

This function solves the Single-Source Shortest Path (SSSP) problem using the Dijkstra algorithm. SSSP solves the shortest path from origin to all other nodes. sssp_dijkstra_solver(.) requires the following inputs:

node The network origin node
network_map The network graph, stored as structure network_map
d An initialised variable of type vector<double> which will contain the shortest paths for each node. This variable is stored in the RoadNetwork class.

build_subnetwork(.)

Note

At present, the elongation ratio for the not-too-long criteria is a globally defined variable set to 1.6. It is planned to allow this value to be user specified.

This function applies the short detour/ not-too-long criteria to the road network to subset the network to only nodes and links which meet the criteria.build_subnetwork(.) requires the following inputs:

network_map The network graph, stored as structure network_map
subnetwork_map An initialised variable of type subnetwork map which will contain the returned subsetted network graph, stored in the RoadNetwork Class
d An initialised variable of type vector<double> which will contain the shortest paths for each node. This variable is stored in the RoadNetwork Class. This variable is initialised by the program if run through the RoadNetwork Class - no user intervention required.

The not-too-long criterion is enforced using an elongation ratio, τra, which typically ranges between 1.3 - 1.6 (this program uses 1.6 by default). The program uses the following mathematical expression to determine the sub-setted links and nodes:

(1 + τra)(lr(heada) − lr(taila)) ≥ la where la is the cost of link a, heada and taila are the head and tail of link a, and lr(heada) and lr(taila) are the cost from the origin r to the head and tail of the link, respectively.

efficient_routes(.)

This function applies the efficient route calculations. It takes the following inputs:

subnetwork_map A variable of type subnetwork map which will contain the subsetted network graph, stored in the RoadNetwork Class
index An integer specifying the starting index of the network nodes (0 or 1)

efficient_routes(.) begins by converting the subnetwork from type subnetwork_map to an adjacency matrix. At current, only one link can exist between two nodes per direction. The adjacency matrix will be of size n × n , where n is the number of unique nodes in the subnetwork. The number of efficient routes is subsequently calculated using the following loop:

    for (int j = 0; j <= number of nodes; j++) {
        for (int m = 0; m <= number of nodes; m++) {
            if (m == j) continue;
            for (int n = 0; n <= number of nodes; n++) {
                if (n == j || n == m) continue;
                LinkMatrix[m][n] += LinkMatrix[m][j] * LinkMatrix[j][n];
            }
        }
    }

The result is a matrix containing the number of efficient routes between the specified origin and all nodes subsetted from the network.

Caution

The required compute increases significantly with additional nodes in the network (algorithm is of order O(n3) ). This will result in long compute times for large networks.

Outputs

The user has the option to save the results of the analysis when the program is initialised through test_initialiser(.). Two outputs are created when the user specifies results to be saved.

write_results(.)

This function uses the JSON for Modern C++ library created by Nlohmann to save the result matrices as into a single JSON file. The JSON format provides flexibility in saving several matrices of different sizes within a single file which cannot easily be achieved using a text file, for example.

The write_results(.) function requires two inputs:

results a 3D matrix of integers comprised of one or more 2D result matrices
output_path a string specifying the output path (including file name and .JSON extension) for the results to be saved into

When the program is called through the test_initialiser(.) function the output_path is set to the same directory as the input adjacency list. The file will be saved as “{input graph name}_Results.json”

write_subnetwork(.)

This function saves the subsetted network created from the build_subnetwork(.) function. It includes the links that meet the not-too-long criteria and is saved in the same format as the input adjacency list [origin destination cost]. The subsetted network is used in the efficient_routes(.). Saving the subnetwork is useful for visualising the results, debugging, or further analysis.

The function requires three inputs:

start_nodes a vector of integers containing the user specified start nodes
subnetwork_maps a vector containing the type subnetwork_map to store one or more subnetworks created from build_subnetwork(.).
output_path a string specifying the output path (including file name and .JSON extension) for the results to be saved into

The subnetworks are saved as a text file. When the program is called through the test_initialiser(.) function the output_path is set to the same directory as the input adjacency list. The file will be saved as “{input graph name}_SubNetwork.txt”
